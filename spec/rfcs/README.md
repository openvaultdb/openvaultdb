# RFCs

## Purpose

Define the proposal area for material architecture or scope changes.

## Key Concepts

- RFC: reviewable proposal that may change specifications, MVP scope, or implementation strategy.
- Sponsor: person responsible for moving a proposal through review.
- Disposition: accepted, rejected, withdrawn, or superseded.

## Normative Requirements

- RFCs MUST be used for cloud sync, Git-backed vault storage, third-party extensions, implicit AI write access, or provider-trust expansion.
- RFCs SHOULD link to affected specifications and test matrix updates.
- Accepted RFCs SHOULD produce decisions or specification changes.

## MVP Behavior

No RFCs are active yet. Fable 5 review may identify the first required RFCs.

## Risks

- RFCs can become a parallel source of truth if specs are not updated.
- Too much process can slow early learning.

## Open Questions

- What template should RFCs use?
- Should security RFCs require a threat-model delta section?

## Acceptance Criteria

- Material scope expansions cannot land through implementation-only changes.
- Accepted RFCs update specifications and tests.
